# Инструкции для ИИ-агентов

Этот репозиторий - стартовый шаблон для HTML-верстки на Pug, SCSS и Vite. При любых изменениях сохраняй существующую структуру, не добавляй лишние фреймворки и не усложняй шаблон без явной необходимости.

## Технологии и команды

- Node.js: `>=20.10.0`.
- Установка зависимостей: `npm install`.
- Dev server: `npm run dev`.
- Production build: `npm run build`.
- Preview production build: `npm run preview`.
- Создание обычного блока: `npm run block -- blockName`.
- Создание блока-миксина: `npm run mixin -- blockName`.

Перед сдачей изменений запускай `npm run build`, если правки затрагивают Pug, SCSS, JS, ассеты или конфигурацию сборки.

## Структура проекта

- `dev/pages` - Pug-страницы. Каждый `.pug` файл становится отдельным `.html` файлом.
- `dev/layouts` - базовые layout-файлы.
- `dev/blocks` - независимые блоки. Для каждого блока создается папка `dev/blocks/blockName` с файлами `blockName.pug`, `blockName.scss`, `blockName.js` по необходимости.
- `dev/blocks/_helpers` - служебные Pug-миксины и helper-файлы.
- `dev/styles/styles.scss` - главный SCSS-файл с подключением helper-стилей.
- `dev/styles/helpers` - переменные, миксины, базовые и утилитарные стили.
- `dev/js` - общий JS. Файлы из этой папки собираются в `main.js`/`main.min.js`.
- `dev/libs` - сторонние CSS/JS, которые собираются в `plugins.css`/`plugins.js`.
- `dev/libs/notConcat` - сторонние CSS/JS, которые не нужно объединять.
- `dev/assets` - проектные ассеты.
- `dev/assets/icons` - SVG-иконки для спрайта `assets/sprite.svg`.
- `dev/images` - контентные/плейсхолдерные изображения.
- `dev/fonts` - шрифты.
- `dev/root` - файлы, которые копируются в корень сборки.
- `public` - результат production build. Не редактируй вручную.

## Форматирование

- Соблюдай `.editorconfig` и `.prettierrc`.
- Используй табы шириной 4 пробела.
- Кодировка: UTF-8, окончания строк: LF.
- Сохраняй финальный перевод строки.
- Не добавляй trailing whitespace, кроме случаев, где это уже разрешено настройками для Pug/Markdown.
- Не меняй стиль кавычек, отступов и переносов в соседнем коде без причины.
- Комментарии добавляй только там, где они действительно объясняют неочевидную логику или важное ограничение. Не комментируй очевидную разметку, классы и базовые CSS-свойства.

## Нейминг

- Для файлов, папок блоков, Pug-миксинов и БЭМ-классов используй camelCase.
- Примеры: `doctorList`, `newsCard`, `siteHeader`.
- Папка блока, Pug-файл, SCSS-файл, JS-файл, имя миксина и корневой класс должны совпадать: `dev/blocks/doctorList/doctorList.pug`, `doctorList.scss`, `doctorList.js`, `mixin doctorList(...)`, `.doctorList`.
- Не смешивай camelCase с kebab-case, snake_case и PascalCase в именах новых блоков.

## Pug

- Страницы в `dev/pages` должны расширять layout:

```pug
extends ../layouts/main.pug

block variables
	- var title = 'Page title'

block content
	.page
		.container.page__container
			h1.page__heading Заголовок
```

- Крупный блок первого уровня на странице обязан содержать контейнер на первом уровне вложенности. Контейнер ограничивает ширину и центрирует контент:

```pug
.about
	.container.about__container
		h1.about__heading Заголовок
```

- Используй существующий `.container` из `dev/styles/helpers/utilitar.scss`.
- Не заменяй `.container` локальными аналогами вроде `.wrap`, `.inner`, `.content`, если нет явной причины.
- Подключай Pug-миксины через `include` рядом с местом использования или в файле, где это принято для конкретной страницы/блока.
- Для SVG-иконок используй helper `+svgElement(name, width, height)` из `dev/blocks/_helpers/_svgElement.pug`.
- Не добавляй инлайн-стили в страницы и блоки. Исключение - служебная sitemap-страница `dev/pages/index.pug`, если она остается технической картой сайта.
- Для больших статичных списков можно заводить массив данных в Pug-переменной рядом со страницей или миксином и рендерить его через `each`, чтобы не дублировать однотипную разметку вручную.

## Доступность

- У контентных изображений должен быть осмысленный `alt` на русском языке.
- У декоративных изображений должен быть пустой `alt=""`.
- SVG-иконки, которые не несут самостоятельного смысла, должны быть скрыты от скринридеров через `aria-hidden="true"`. Helper `+svgElement` уже делает это по умолчанию.
- Если внутри кнопки или ссылки нет видимого текста, добавляй `aria-label` с понятным описанием действия на русском языке.
- Интерактивный элемент должен соответствовать смыслу действия: для перехода используй ссылку, для действия на странице - кнопку.

## БЭМ

- Используй БЭМ для HTML-классов и SCSS.
- Блок: `.block`.
- Элемент: `.block__element`.
- Модификатор: `.block--modifier` или `.block__element--modifier`.
- Не вкладывай элементы одного блока в элементы другого блока как часть имени. Неверно: `.card__title__icon`.
- Не стилизуй блок через контекст страницы без необходимости. Предпочитай самодостаточные классы блока.
- Не завязывай стили на теги, id или глубокую DOM-структуру, если можно использовать БЭМ-класс.
- Служебные классы из `utilitar.scss` можно использовать как утилиты, но не подменяй ими структуру блока.

## Блоки и миксины

- Если блок используется в нескольких местах или потенциально может переиспользоваться, делай его Pug-миксином через `npm run mixin -- blockName`.
- Если блок нужен только один раз и не предполагает переиспользования, допустим обычный блок через `npm run block -- blockName`.
- Папка блока должна называться так же, как сам блок: `dev/blocks/newsList/newsList.pug`, `dev/blocks/newsList/newsList.scss`, `dev/blocks/newsList/newsList.js`.
- Не создавай глобальные стили блока в `dev/styles/helpers`. Стили конкретного блока держи в его `blockName.scss`.
- Сборка сама находит SCSS и JS в `dev/blocks`, поэтому не добавляй ручные импорты блоков в `styles.scss` без отдельной причины.
- Файлы блоков собираются в лексикографическом порядке пути. Не рассчитывай на порядок CSS/JS между независимыми блоками.

## Работа с Figma-компонентами

- Не переноси Figma-компоненты в HTML один к одному автоматически.
- Делай отдельный Pug-миксин/БЭМ-блок только для компонентов, которые реально переиспользуются в верстке или являются самостоятельной сущностью.
- Если Figma-компонент создан только для удобства дизайнера и используется внутри одной секции, верстай его как элемент родительского блока.
- Варианты Figma-компонента переноси в БЭМ-модификаторы, если структура остается общей.
- Если визуально похожие Figma-компоненты имеют разный смысл, разные данные или разное поведение, не объединяй их преждевременно.
- Если Figma-компонент содержит другой компонент, не копируй вложенность автоматически. В HTML важнее смысловые границы блоков и чистая БЭМ-структура.

Примеры:

- Карточка новости, карточка врача, кнопка, таб, хлебная крошка или аккордеон, которые повторяются в разных местах, обычно становятся отдельными блоками/миксинами: `newsCard`, `doctorCard`, `baseButton`.
- Figma-компонент `TextBlock`, который используется только внутри секции `about`, может быть элементом родительского блока: `.about__text`, а не отдельным блоком.
- Варианты одной кнопки лучше делать модификаторами: `.baseButton--primary`, `.baseButton--secondary`.
- Похожие карточки новостей, врачей и услуг не нужно автоматически объединять в один блок, если у них разные данные, ссылки, hover-состояния или будущая логика.

Рекомендуемый шаблон миксина:

```pug
mixin news(modificator, extraClasses, title, lead, count)
	- classList = 'news'

	if modificator
		each mod in modificator
			- classList = classList + ' news--' + mod

	if extraClasses
		- classList = classList + ' ' + extraClasses

	div(class=classList)
		.container.news__container
			if title
				h2.news__title #{title}

			if lead
				p.news__lead #{lead}

			include ../newsItem/newsItem.pug

			.news__list
				- var i = 0
				while i < count
					- i = i + 1
					+newsItem()
```

Правила для параметров миксинов:

- Первые параметры для переиспользуемых блоков: `modificator`, `extraClasses`.
- `modificator` ожидается как массив строк и добавляет классы вида `block--mod`.
- `extraClasses` добавляется к корневому классу как строка.
- Заголовки, вводный текст, подписи, состояния, количество элементов и другие изменяемые данные передавай параметрами.
- Списки карточек/новостей/элементов можно захардкодить внутри миксина, если шаблон используется для статической верстки и пользователь не просит иное.
- Если внутри миксина нужен другой переиспользуемый блок, подключай его через `include` и вызывай его миксин.

## SCSS

- Верстка ведется mobile first: базовые стили пишутся для мобильной ширины, затем расширяются через `min-width`.

```scss
.about {
	padding: 32px 0;

	@media (min-width: $tablet) {
		padding: 48px 0;
	}

	@media (min-width: $desktop) {
		padding: 72px 0;
	}
}
```

- Используй актуальные брейкпоинты и размеры контейнера из `dev/styles/helpers/variables.scss`. В начале проекта пользователь может изменить эти SASS-переменные под конкретный макет; считай их источником правды.
- Не хардкодь ширину контейнера и основные брейкпоинты внутри блоков, если нужное значение уже есть в `variables.scss`.
- Цвета из макета, если они оформлены как переменные/токены, должны быть CSS-переменными в `dev/styles/helpers/variables.scss`.
- Для удобства можно завести SCSS-переменную, которая ссылается на CSS-переменную:

```scss
$white: var(--white);

:root {
	--white: #ffffff;
}
```

- Не добавляй новые цветовые значения прямо в блок, если цвет является частью палитры проекта. Сначала добавь CSS-переменную в `variables.scss`.
- Локальные одноразовые значения допустимы только для действительно локального оформления, которое не является дизайн-токеном.
- Не используй `!important`, кроме утилит или вынужденного переопределения внешней библиотеки.
- Не добавляй CSS reset: базовые стили уже есть в `common.scss`.
- Шрифты подключай через миксин `webfont` в `fonts.scss`.

## JavaScript

- Общий JS находится в `dev/js`.
- JS конкретного блока держи в `dev/blocks/blockName/blockName.js`.
- Сборка сама объединяет JS из `dev/js` и `dev/blocks`; ручные импорты для обычных блоковых скриптов не нужны.
- Не добавляй зависимости без явной необходимости.
- Переиспользуй существующие helper-функции, например `sendData` из `dev/js/common.js`.
- Пиши код так, чтобы отсутствие блока на странице не ломало скрипт:

```js
const block = document.querySelector(".block");

if (block) {
	// block logic
}
```

## Ассеты

- SVG-иконки для спрайта клади в `dev/assets/icons`.
- Декоративные изображения, которые используются именно в верстке, клади в `dev/assets`.
- Контентные изображения и заглушки, которые могут измениться после натяжки на CMS/бэкенд, клади в `dev/images`. Примеры: картинка новости, фото сотрудника, изображение карточки товара.
- Изображения из `dev/images`, как правило, можно заменить плейсхолдером `https://placehold.jp/150x150.png`, где `150x150` нужно заменить на фактический размер картинки.
- Шрифты клади в `dev/fonts` и подключай через `fonts.scss`.
- Не редактируй файлы в `public`: они генерируются сборкой.

## SEO и head

- Базовый `<head>` описан в `dev/blocks/_head/_head.pug`.
- Для страницы задавай `title` в `block variables`.
- Если меняешь SEO-мета, делай это централизованно в `_head.pug` или через параметры/переменные layout, а не копированием head на каждую страницу.

## Проверка изменений

Проверяй не весь проект подряд, а область, которую затрагивает задача.

- Если задача была сверстать или исправить отдельную страницу, проверяй эту страницу и общие файлы, которые были изменены для нее.
- Если задача была сверстать или исправить блок/миксин, проверяй сам блок/миксин и все страницы/блоки, где он подключен.
- Если задача затрагивает общий layout, `_head.pug`, helper-миксины, `variables.scss`, `common.scss`, `utilitar.scss`, сборочные скрипты или общий JS, расширяй проверку на страницы и блоки, которые могут зависеть от этих изменений.
- Если была создана новая страница, не нужно проверять все старые страницы без причины. Достаточно проверить новую страницу, ее подключенные блоки и измененные общие файлы.
- Если пользователь дал доступ к Figma MCP или явно указал макет, в самом конце сверь реализованную страницу/компонент с макетом, если пользователь не попросил этого не делать.

Минимальный чеклист для затронутой области:

1. Убедись, что новые файлы лежат в правильных папках.
2. Проверь БЭМ-именование классов.
3. Проверь, что крупные top-level блоки на страницах содержат `.container`.
4. Проверь mobile-first порядок SCSS.
5. Проверь, что новые цвета-токены добавлены в `variables.scss` как CSS-переменные.
6. Запусти `npm run build`, если правки затрагивают Pug, SCSS, JS, ассеты или конфигурацию сборки.

Если build не запускался или упал, явно сообщи об этом и укажи причину.
